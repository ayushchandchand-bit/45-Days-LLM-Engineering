We can already call an LLM. But real apps need prompts reused, steps chained, data validated, memory kept — and eventually tools and agents. LangChain is the framework that makes all of that snap together.

Why we need a framework

Raw API calls don't scale into apps

A single client.chat.completions.create(...) is fine for a demo. But a real product needs the same plumbing over and over — and you end up rewriting it by hand.

😓 Doing it by hand

Glue code for every prompt, provider locked in, JSON parsed manually, memory re-invented, no shared way to add tools.

🧩 With a framework

Reusable prompts, one interface for any model, typed outputs, built-in memory & retrieval, and a standard path to tools + agents.

The shift

Stop writing plumbing, start writing logic. LangChain gives every LLM the same shape so the pieces above yours all compose.

The one big idea

LangChain = one interface + composable pieces

Every component — a model, a prompt, a parser, a retriever, a tool — is a Runnable: it has .invoke() and snaps to the next piece with the | operator.

Remember this one line

Build messages → invoke → get a reply. Same shape for every model, every step.

The core · LCEL

Chaining with the | operator

LCEL ("LangChain Expression Language") is one rule: the output of the left flows into the right. Read it left to right.

prompt

fill the blanks
|

model

the LLM answers
|

parser

clean text / data
chain = prompt | model | parser chain.invoke({"question": "What is Python?"}) # -> a clean string

Same chain also gives you .batch([...]) (many at once) and .stream(...) (live typing) for free.

Reuse & structure

Templates in, typed data out

Prompt template — write once, reuse
prompt = ChatPromptTemplate.from_messages([ ("system", "Translate into {language}."), ("human", "{text}"), ])
Structured output — real objects
class Movie(BaseModel): title: str rating: int model.with_structured_output(Movie)
Why it matters

Templates keep roles & catch missing variables; with_structured_output returns a validated Pydantic object — no fragile JSON parsing. Text for humans, data for your code.

Conversation memory

Remembering the chat

An LLM is stateless — it forgets. You carry a history list and inject it into the prompt with a placeholder, so each call sees the whole conversation (Day 16, now in LangChain form).

prompt = ChatPromptTemplate.from_messages([ ("system", "You are helpful."), MessagesPlaceholder("history"), # <-- past turns land here ("human", "{input}"), ]) # after each turn: history.append(HumanMessage(...)); history.append(AIMessage(...))
Latest-syntax note

RunnableWithMessageHistory is deprecated in LangChain 1.x. For simple chat, keep the list yourself; for heavier state, that job moves to LangGraph.

The big leap

From a chain to an agent

A chain runs a fixed path you wrote. An agent lets the model decide what to do next — which tool to call, and when it's done. It's an LLM in a loop with tools.

🔗 Chain

You fixed the steps: prompt → model → parser. Predictable, no decisions.

🕹️ Agent

The model picks steps: "I need to search… now calculate… now answer." Dynamic, tool-using.

Definition to remember

Agent = LLM + tools + a loop that keeps going until the task is done.

How an agent thinks

The reason–act loop

Think → Act → Observe, over and over, until the model decides it can Answer. This is the ReAct pattern.

What agents act with

Tools: giving the model hands

A tool is just a Python function the model is allowed to call. You describe it; the model decides when to use it and with what arguments.

from langchain_core.tools import tool @tool def get_weather(city: str) -> str: """Get the current weather for a city.""" # <- the model reads this return lookup(city) model.bind_tools([get_weather]) # now the model can request it
The docstring is the API

The function name, type hints, and docstring are how the model knows what the tool does and how to call it. Clear docstrings = reliable tool use.

When a chain isn't enough

Chains are lines; agents need graphs

A straight | chain can't loop or branch. Once you need "keep calling tools until done", you need a state machine — that's LangGraph.

➡️ LCEL chain

A fixed line of steps. Perfect for prompt→model→parser and RAG. No loops.

🔄 LangGraph

Nodes + edges with state. Loops, branches, tool cycles, persistence. The runtime real agents run on.

In LangChain 1.x

Agents are built with create_agent and run on LangGraph under the hood. Simple flow → a chain. Stateful, tool-looping agent → LangGraph. That's Day 22.
