How teams build together
without stepping on each other

The final piece: branches, merges, and pull requests — the workflow behind every real software team and every open-source project.

This is also how you'll contribute to projects and show teamwork to employers.
The problem branches solve

Everyone editing the same code = chaos

Imagine five people typing into the same document at once, live. Someone's half-finished feature breaks everyone else. You're scared to experiment because you might break the working version.

Branches fix this. Each person (or each feature) gets a safe, isolated copy to work in — and the main version stays stable.

The key concept

A branch = a parallel copy of your code

🌳 main

The official, always-working version. You protect this one.

🌿 a feature branch

A safe sandbox split off from main. Experiment freely — if it breaks, main is untouched.

Analogy: main is the published book. A branch is a working draft — you edit the draft, and only merge it back when it's good.

Branch, work, merge — visually

The branch & merge graph

Split off main → commit freely on the branch → merge it back when it's ready. main was never at risk.

Doing it in commands

Create & switch to a branch

$ git switch -c feature/login # create a branch & jump onto it
# ...edit files, then the normal loop:
$ git add .
$ git commit -m "Add login form"
$ git push -u origin feature/login # push the branch to GitHub

$ git switch main # hop back to main anytime

Your edits on the branch don't touch main until you merge. Switch between them freely.

Bringing the work back

Merging the branch into main

When the feature is done and tested, merge combines its commits back into main:

$ git switch main # stand on the branch you're merging INTO
$ git merge feature/login # pull the feature's work in

In practice on GitHub you rarely type this — you click "Merge pull request" (next slides). Same idea, with a review step.

Don't panic when this happens

Merge conflicts

If two branches changed the same lines, Git can't decide who wins — so it asks you. It marks the spot:

<<<<<<< HEAD
greeting = "Hello" # your version
=======
greeting = "Hi there" # the other branch's version
>>>>>>> feature/login

Just edit the file to keep what you want, delete the <<< === >>> markers, then add + commit. Conflicts feel scary once, then never again.

The heart of GitHub collaboration

Pull Requests (PRs)

A Pull Request says: "I finished my branch — please review and merge it." It's where teammates comment, review, and approve code before it joins main.

1branch→ 2commit→ 3push→ 4open PR→ 5review→ 6merge

PRs are how every team and open-source project ships. "I have 20 merged PRs" is a real signal to employers.

Contributing to code you don't own

Fork → branch → PR

🍴 Fork

Make your own copy of someone else's repo on GitHub.

🌿 Branch & fix

Clone your fork, make a branch, fix a bug or add a feature.

📨 Pull Request

Open a PR back to the original. Maintainer reviews & merges.

This is open source. Even one merged PR to a real project is a portfolio gold-star — proof you can work in someone else's codebase.

Putting it all together

The everyday team workflow

  • git pull on main — start from the latest.
  • git switch -c feature/x — branch off for your task.
  • Work → add → commit (repeat) → push the branch.
  • Open a Pull Request → teammates review → merge into main.
  • Everyone pulls the updated main. Repeat.

This single loop runs at basically every software company on earth.
