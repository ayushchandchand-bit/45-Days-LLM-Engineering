Embeddings:
turning meaning into numbers

How does a computer find the right note out of thousands when none of them share your exact words? It stops matching letters and starts matching meaning. That trick is an embedding.

Text in → numbers out → similar meaning sits close together.
Why we need this

Keyword search matches letters, not meaning

You search your notes for "motor vehicle". A note says "the automobile won't start" — clearly relevant. But it shares zero words with your search, so plain keyword search misses it.

🔤 Keyword search

Matches if the same letters appear. "automobile" ≠ "motor vehicle" → no match. Misses synonyms, paraphrases, typos.

🧠 Semantic search

Matches if the meaning is close. "automobile" ≈ "motor vehicle" → strong match. This is what embeddings unlock.

The whole phase rests on this

Search, RAG, document Q&A (your next project) — all of it is "embed everything, then find the nearest vectors." Today is the foundation.

The one big idea

An embedding = text turned into a list of numbers

An embedding is a list of numbers (a vector) that stands in for a piece of text. The magic is how the numbers are chosen:

Remember this one line

Similar meaning → similar numbers → close together in space.

"I love AI" → -0.01+0.07+0.02 -0.05+0.09… 384 numbers

That row of numbers is the embedding. You won't read meaning in them — but the computer can compare them.

Build the intuition

Think of meaning as a map

Pretend we describe each word with just 2 numbers:

  • how animal is it? (left→right)
  • how vehicle is it? (bottom→top)

Now every word is a point. cat and dog land together; car and truck land together — far away. Close on the map = close in meaning.

Module 01 builds exactly this with 2 hand-made numbers — no libraries, pure Python.

From toy to real

Real text needs more than 2 features

We hand-picked "animal-ness" and "vehicle-ness" just to see the idea. Real language has countless shades of meaning — is it formal? past tense? about money? happy?

✋ Our toy

2 features, chosen by hand. Good for a picture, useless for real text.

🤖 A trained model

Picks hundreds of features for you, learned from billions of sentences.

📏 Our model today

all-MiniLM-L6-v2 → 384 numbers per text. Free, runs on your laptop.

Same idea, more dimensions

2 numbers = a point on a flat map. 384 numbers = a point in "384-dimensional space." We can't picture it — but the math of "who's nearest" works exactly the same.

Module 04 · the measure

How do we measure "close"?

We compare the direction two vectors point. Same direction = same meaning. This is cosine similarity — it looks at the angle, ignoring how long the arrows are.

Don't fear the formula

Cosine similarity, in plain words

cosine(a, b) = (a · b) ----------- |a| × |b|

In numpy it's one line:

np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))
  • a · b (dot product) — multiply matching numbers, add them all up.
  • |a| (norm) — the arrow's length: sqrt(sum of squares).
  • Divide by both lengths so only direction matters.
Shortcut you'll use a lot

Our model returns unit-length vectors (length = 1). Then the dividing does nothing, so cosine = just the dot product. That's why vector databases are fast.

Reading the score

One number, from −1 to +1

In practice with this model

Related text scores roughly 0.3 – 0.9; unrelated text scores near 0. Compare scores within the same set to find the best match — a single score isn't a "grade out of 1".

Module 05 · providers

Local model vs cloud API

We default to a free, local model. A cloud API (like Gemini) does the same job — the code shape is identical, only the line that makes the vector changes.

💻 Local · sentence-transformers

Free, private (text never leaves your machine), works offline. 384 numbers. Our default all course.

☁️ Cloud · Gemini

Free tier then paid, needs internet + an API key, sends text to Google. Bigger vectors (768–3072). Optional.

The real lesson

The provider is swappable — the retrieval math (cosine, nearest-neighbour) is exactly the same either way.
